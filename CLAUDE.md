## First Action — ALWAYS

Read `ai-sdlc/tracking/project-state.md` before doing anything.

## Commands

All commands are slash commands via `.claude/commands/ai-*.md`. Type `/ai-` and Tab, or `/ai-help` for the full list (37 slash commands — see `docs/framework-reference.md`).

Each slash command reads its definition from `ai-sdlc/commands/ai-{name}.md`. Command files contain: `reads:`, `updates:`, `requires_approval:`, `token_budget:`, `phase_constraint:`.

## How to Work

1. Read `ai-sdlc/tracking/project-state.md` — know where we are.
2. Read `ai-sdlc/playbooks/_index.md` — know your roles.
3. Read the relevant phase file under `ai-sdlc/phases/` — know current activities.
4. Execute using the appropriate command.
5. Update `ai-sdlc/tracking/project-state.md` after any state change.
6. Run `/ai-sync` to regenerate derived files.

## File Paths

All ai-sdlc files are under `ai-sdlc/`. When commands reference `tracking/project-state.md`, the actual path is `ai-sdlc/tracking/project-state.md`.

## Tracking Rules

**Source-of-truth** (commands write to): `tracking/project-state.md`, `tracking/story-index.md`, `tracking/changelog.md`, `tracking/decision-log.md`, `tracking/risks.md`, `tracking/anti-patterns.md`.

**Derived** (regenerated by `/ai-sync`, NEVER write directly): `tracking/dashboard-data.json`, `tracking/sprint-backlog.md`, `tracking/rtm.md`.

**Session:** `tracking/recovery.md` — build checkpoint.

## Token Efficiency

1. **Frontmatter-first**: Read YAML frontmatter before full file. Only expand if relevant.
2. **Follow `token_budget`**: small = frontmatter only, medium = selective, large = full reads.
3. **Conditional reads**: Only read files for the selected mode/flag, not all `reads:`.
4. **Cache in loops**: In multi-step commands, read shared context ONCE at start.
5. **Dashboard sync**: NEVER write `dashboard-data.json` directly. Update source-of-truth, then `/ai-sync`.

## Approval Rules

See `ai-sdlc/config/workflow-rules.yaml`. Quick: reads + in-sprint code = autonomous. Adding stories, architecture changes, gate passage, phase transitions = human approval.

## Session Recovery

If interrupted: read `ai-sdlc/tracking/recovery.md` -> read `project-state.md` -> resume from checkpoint -> clear recovery when done.

## Evidence Packages

Quality gates verify canonical evidence package completeness before phase transitions:

- **Discover-to-B0**: charter, requirements, architecture-package, observability-intent, b0-readiness-checklist
- **Ship S1–S3**: s1-validation-evidence, s2-hardening-evidence, s3-deploy-readiness
- **Evolve**: feedback-log, iteration-plan, tech-debt-register, performance-review

ANY-phase commands (`ai-status`, `ai-help`, `ai-risk`, `ai-change`, `ai-handoff`) read this package readiness to provide informed guidance.

## Generated Skills

Run `/ai-scaffold skills` to create project-specific pattern overlays in `ai-sdlc/playbooks/generated/`.

- Most generated overlays are strongest after B0 (Walking Skeleton).
- Observability should start with Discover intent (`docs/design/observability-intent.md`) and then be refreshed after B0 so the generated constraints reflect proven evidence.

Typical generated files:

- `db-constraints.md`, `api-constraints.md`, `design-tokens.md`, `ui-constraints.md`.
- Observability (four files): `logging-constraints.md`, `tracing-constraints.md`, `metrics-constraints.md`, `apm-integration.md`.

These skills are loaded by `/ai-build`, `/ai-preflight`, `/ai-review`, `/ai-secure`, `/ai-test`, `/ai-wire`, `/ai-harden`, `/ai-monitor`, `/ai-deploy`, `/ai-debt`, `/ai-story`, and `/ai-ui` during B1-S4 to keep new code pattern-consistent.
